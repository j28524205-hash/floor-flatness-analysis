### Phase별 핵심 변경사항 (왜 고쳤는지)

**Phase 1~2**는 최초 발견된 크래시 버그들 — 파일 경로 오류, PDF 폰트 하드코딩(Windows 전용이라 이식성 문제), 소규모/퇴화 데이터에서의 크래시(`remove_outliers`가 표준편차 0일 때 전체 삭제, `fit_plane_ransac`이 평면을 못 찾으면 `None` 접근으로 크래시, `griddata`가 일직선 데이터에서 크래시), 판정 기준과 RANSAC 임계값이 뒤섞여 있던 문제, 업로드 검증 부족을 해결함.

**Phase 3**은 "다른 리뷰 도구가 옛날 버전 파일을 검사해서 이미 고친 문제를 다시 보고"하는 사고를 겪은 후 만든 동기화 체계. **코드를 고칠 때마다 세 곳(또는 mockup은 두 곳)을 전부 동기화해야 한다는 규칙이 이때 확립됨.** 이후 모든 Phase는 매 수정 후 3곳(mockup은 2곳) `diff` 확인을 검증 절차에 포함.

**Phase 4**는 RANSAC의 근본적인 정확도 문제(3점 샘플 평면을 그대로 쓰던 것을 인라이어 전체 최소제곱 재적합으로 개선)와 구조 정리(분석 로직을 `run_analysis()` 단일 함수로 추출 — 이전엔 `try` + `if True:` 트릭으로 들여쓰기를 억지로 맞췄던 것을 정리함).

**Phase 5~8**은 여러 차례의 외부/자체 코드 리뷰를 반복하며 점진적으로: RANSAC 신뢰도 검증(최소 인라이어 비율), 안전장치 강화, 파라미터를 코드 상수에서 UI 조정 가능하도록 노출, 각종 진단 정보(이상치 제거 비율, 보간 방식, 좌표 단위 이력)를 화면과 보고서에 투명하게 기록하는 방향으로 개선함. **이 과정에서 "바닥 분리 OR 기본값을 AND로 바꿔야 하지 않냐"는 지적이 4번, "크랙 점수를 절대 기준으로 바꿔야 하지 않냐"는 지적이 여러 번 반복됐는데, 둘 다 논문에 서술된 알고리즘과의 일치성을 이유로 변경하지 않기로 확정함.** (이 결정을 다음 세션에서 뒤집으려면, 논문이 이미 이 알고리즘 설명을 전제로 제출 준비 중이라는 점을 먼저 고려해야 함 — `한일심포지엄_영문초안.md` 참고.)

**Phase 9**는 코드가 아니라 PRD 5절(기능 요구사항)을 시각적으로 재현한 **정적 HTML 목업**. 실제 앱과 연동되지 않으며, 표시된 수치(총 입력점 45,231개, A/B1/B2/C 87.1%/4.2%/6.9%/1.8%, 불량률 12.9%)는 전부 가상값 — 논문 예시 통계와 의도적으로 맞춘 것. 디자인은 블루프린트/테크니컬 엔지니어링 톤(어두운 네이비 + 청록 격자, `Big Shoulders`/`IBM Plex Mono` 폰트). 상세 내용은 `평활도PRD.md` 12절 참고.

**Phase 10**은 실제 배포. `RISE 학문후속세대 연구` 폴더를 git 저장소로 만들고(대용량 pptx·행정문서는 `.gitignore`) GitHub 공개 저장소에 push한 뒤, Streamlit Community Cloud와 연결함. 배포 과정에서 실제로 두 가지 버그를 잡았는데 둘 다 **로컬 환경과 배포(Linux 컨테이너) 환경의 차이**에서 비롯된 것: ① `requirements.txt`에 로컬 버전을 `==`로 그대로 박아넣었더니 `pandas==3.0.3`이 요구하는 `numpy` 최소버전과 안 맞아 설치 자체가 실패 — 버전을 `>=`로 완화해서 해결. ② `kaleido>=1.0`이 PDF용 이미지 렌더링에 시스템 Chrome을 요구하는데 배포 서버엔 Chrome이 없어서 PDF 생성이 실패 — `kaleido==0.2.1`(자체 Chromium 번들)로 내려서 해결. **이 두 번째 버그가 "PDF 생성 실패" 경고로 명확하게 사용자에게 표시된 것은 Phase 3에서 고친 `insert_fig` 예외처리(조용히 무시하지 않고 에러를 올리게 한 것)가 실전 배포 환경에서 그대로 효과를 본 사례.** 배포 URL과 GitHub 링크는 위 "프로젝트 개요" 참고. 이어서 10.8에서, 원래 "계획 없음"이었던 목업 스타일 반영을 **사용자가 배포된 앱을 직접 보고 나서 요청**해 진행함 — `.streamlit/config.toml`(다크 테마 색상) + 전역 CSS 주입(Google Fonts, 메트릭/버튼/입력창/탭 스타일) + Plotly `pio.templates.default`로 전체 차트를 다크 테마로 통일(개별 figure 코드는 안 건드림). Chrome `--headless --screenshot`으로는 Streamlit의 WebSocket 렌더링이 안 끝난 채로 캡처돼 빈 화면만 나왔고, **Playwright(`networkidle` 대기)로 바꿔서 검증**함 — 이 앱을 스크린샷으로 확인할 일이 또 있으면 Playwright를 쓸 것.

마지막으로 10.9는 **사용자가 배포된 앱을 실제로 써보다가 발견한 버그**: 분석 실행 후 3D 시각화의 "Z 과장 배율" 슬라이더를 움직이면 화면이 "분석 실행 누르기 전"으로 돌아가 보였음. 원인은 Streamlit의 재실행(rerun) 모델 — `st.button()`은 클릭된 딱 그 순간의 재실행에서만 `True`이고, 슬라이더 조작처럼 다른 위젯이 트리거한 재실행에서는 다시 `False`가 됨. 결과 표시 코드 전체가 `elif st.button(...):` 안에 있었기 때문에, 슬라이더를 만지면 그 블록째로 실행이 스킵돼 결과가 사라진 것처럼 보인 것. **이런 종류의 버그는 Streamlit 앱에서 매우 흔한 패턴이니, 앞으로 "버튼 클릭 후 나온 결과를 다른 위젯으로 계속 조작해야 하는 화면"을 새로 만들 때는 처음부터 `st.session_state`에 결과를 저장하는 방식으로 설계할 것.** 수정은 `st.session_state.analysis_result`에 저장하고 그 존재 여부로 표시 블록을 게이팅하는 방식(디스플레이 코드 자체의 들여쓰기는 그대로 두고, 감싸는 조건문만 바꿔서 대규모 재작성 없이 처리함).

10.11은 **가장 심각했던 실사용 버그**: 사용자가 실제 스캐너로 취득한 대규모 샘플 데이터(수십만~수백만 점)를 넣고 분석했더니 앱이 중간에 재시작됨. `separate_floor`/`detect_cracks`가 포인트마다 Python 반복문으로 SVD/고유값분해를 하던 구조라, 대규모 데이터에서 Streamlit Cloud 무료 티어의 메모리 한도(~1GB)를 넘기거나 응답이 너무 오래 걸려 강제 재시작된 것으로 추정됨. 이건 사실 8차 이전부터 "알려진 제한사항"으로 문서에 적혀만 있던 항목(11.2)이었는데, **실제 장애가 나고 나서야 우선순위가 올라가 처리함** — 다음에 비슷하게 "언젠가 고치면 좋음"으로 남겨둔 성능 이슈가 있으면, 실사용 규모를 미리 가늠해보고 더 일찍 처리하는 게 나을 수 있다는 교훈. 수정은 두 함수의 포인트별 반복문을 `np.einsum` + 배치 `np.linalg.eigh`로 벡터화(수정 전후 결과가 오차 1e-11 수준으로 완전히 일치함을 먼저 확인한 뒤 교체함 — 이런 수치 알고리즘 리팩터링은 항상 이렇게 등가성 검증부터 하고 진행할 것). 추가로 `MAX_ANALYSIS_POINTS`(기본 30만) 상한을 두어, 벡터화로도 감당하기 버거운 초대형 데이터는 시드 고정 무작위 다운샘플링 후 분석하도록 안전장치를 뒀다.

10.12는 10.9 버그의 재발이었다. 10.9에서 "슬라이더 조작 시 결과 사라짐"을 고쳤다고 생각했는데, 사용자가 대규모 데이터를 **실제로 업로드**해서 분석한 뒤 슬라이더를 움직이니 또 사라졌다고 보고함. 원인은 "새 데이터 로드 시 `analysis_result` 초기화" 코드가 `if uploaded:` 블록 안에 있었기 때문 — `st.file_uploader()`는 한번 업로드하면 그 파일을 계속 들고 있어서 다른 위젯이 유발한 재실행에서도 이 블록이 매번 재실행되고, 그때마다 결과를 초기화해버렸다. **10.9 검증 때 "샘플 데이터 사용" 버튼으로만 테스트했던 게 문제** — 버튼(`st.button()`)과 업로더(`st.file_uploader()`)는 재실행 시 반환값 유지 방식이 다르다는 걸 놓쳤음. **교훈: 세션 상태 버그를 고칠 땐 데이터가 들어오는 모든 경로(버튼 클릭, 파일 업로드 등)를 각각 재현 테스트해야 하며, 한 경로에서 확인됐다고 다른 경로도 같다고 가정하면 안 된다.** 수정은 업로드 파일의 서명(`file_id` 또는 `name`+`size`)을 저장해두고 실제로 바뀐 경우에만 재처리하는 방식.

### 아직 결정 안 된 것 (Phase 11 백로그)
- **k=10/15/20 (KNN 이웃 수), 균열 점수 가중치 40/40/20%, `MIN_POINTS=30`**: 8차까지 UI로 노출한 파라미터 목록에서 빠져 있음. 노출할지, 코드 상수로 유지할지 아직 논의 안 됨.
- **3D 시각화 서브샘플링(`vis_idx = np.random.choice(...)`) 시드 미고정**: 낮은 우선순위 항목으로 여러 차례 리뷰에서 지적됐지만 아직 수정 안 함. 분석 결과 자체엔 영향 없고 화면에 보이는 점만 매번 달라짐.
- **실측 데이터 검증**: 지금까지 모든 검증은 (a) 논문용 가상 데이터(별도 스크립트) 또는 (b) 앱 자체의 함수 단위 테스트(대칭 링, 퇴화 데이터 등) 뿐. 실제 스캐너로 취득한 점군으로 검증한 적 없음.
- **웹 배포**: Streamlit은 상시 서버가 필요해 Firebase Hosting과 안 맞는다고 이미 결론 냄. Cloud Run 또는 Streamlit Community Cloud 후보로 논의됐으나 아직 미착수.
- ~~목업(Phase 9)을 실제 앱에 반영할지~~: **완료됨 (Phase 10.8)**. 원래 "계획 없음"이었으나 사용자가 배포 후 직접 요청해서 진행함 — `.streamlit/config.toml` + CSS 주입 + Plotly 다크 템플릿. 향후 Streamlit 버전이 크게 바뀌면 `data-testid` 기반 CSS 선택자가 깨질 수 있으니, 스타일이 갑자기 어긋나 보이면 이 부분(파일 상단 CSS 블록)부터 확인할 것.
- **다양한 마감재 유형 일반화 / 실시간 연속 모니터링**: PRD 3절 Out of Scope에 명시된 항목. 별도 논의 없이는 착수하지 않음 (참고용으로만 기록).
